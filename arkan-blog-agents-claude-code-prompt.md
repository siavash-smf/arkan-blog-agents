# پرامپت ساخت سیستم مولتی‌ایجنت تولید بلاگ‌پست آرکان (فاز ۳) — برای Claude Code

> این پرامپتی است که با آن پروژه‌ی [`arkan-blog-agents`](https://github.com/siavash-smf/arkan-blog-agents) ساخته شد. این فاز سوم پروژه‌ی «شرکت آرکان» است (فاز ۱: وب‌سایت، فاز ۲: چت‌بات RAG). همان استک آشنا و همان کلید OpenRouter فاز ۲ استفاده می‌شود.
>
> برای بازتولید، این فایل را همراه `arkan-client-brief.md` و `arkan-brand-guide.md` در ریشه‌ی یک پروژه‌ی Next.js تازه قرار بده و متن زیر را به Claude Code بده.

## نقش و هدف

تو یک مهندس ارشد هوش مصنوعی و متخصص طراحی سیستم‌های **مولتی‌ایجنت** هستی. قرار است یک سیستم واقعی بسازی که کل چرخه‌ی تولید محتوای بلاگ شرکت «آرکان» را خودکار کند: از **پیدا کردن ایده** تا **پژوهش، نگارش، ویرایش، سئو، انتشار** و — مهم‌تر از همه — **بهبود خودش در طول زمان**.

این یک نمونه‌ی آموزشی است (محتوای هفته‌ی ششم دوره‌ی هوش مصنوعی). پس هدف فقط «کار کردن» نیست؛ کد باید **الگوهای مهندسی مولتی‌ایجنت** را شفاف و قابل‌توضیح نشان دهد. کامنت‌های آموزشی فارسی در نقاط کلیدی بگذار تا دانشجو «چرا»ی هر تصمیم را بفهمد.

**پیش از شروع:** دو فایل `arkan-client-brief.md` و `arkan-brand-guide.md` را بخوان. هویت، لحن و مخاطب همه‌ی ایجنت‌ها باید از همین‌ها بیاید.

**فلسفه‌ی معماری — «تقسیم کار بین متخصص‌ها، مثل یک تحریریه‌ی واقعی».**
یک LLM با یک پرامپت غول‌پیکر («یک مقاله‌ی سئوشده بنویس») خروجی متوسط می‌دهد، چون هم‌زمان چند شغل متفاوت (ایده‌پردازی، پژوهش، نگارش، نقد، سئو) را انجام می‌دهد و هر شغل مهارت، لحن و دمای متفاوتی می‌خواهد. راه‌حل: هر شغل را به یک **ایجنت متخصص** بسپار — یک LLM با system prompt تخصصی، قرارداد خروجی صریح، و تنظیمات مخصوص خودش.

## استک فنی

- **Next.js 14+** (App Router) + **TypeScript** — همان استک فاز ۱ و ۲
- **AI SDK (`ai`)** + **`@ai-sdk/openai-compatible`** — همه‌ی مدل‌ها از طریق **OpenRouter** با یک کلید واحد (`OPENROUTER_API_KEY`)، base URL `https://openrouter.ai/api/v1`. تعویض مدل = تغییر یک slug.
- **Zod** برای قرارداد خروجی ایجنت‌ها
- **Supabase** برای ذخیره‌سازی دائمی (اختیاری در حالت یادگیری — پایین توضیح داده شده)
- **`marked`** برای رندر مارک‌داون مقاله
- **Tailwind CSS** + RTL کامل، فونت‌های استعداد/وزیرمتن، رنگ‌های برند آرکان
- آماده‌ی استقرار روی **Vercel** با یک **کرون هفتگی**

> **درس فاز ۲ را تکرار کن:** برخی مدل‌های OpenRouter استدلال اجباری دارند و ممکن است کل بودجه‌ی توکن را صرف reasoning کنند. در هسته‌ی AI، روی همه‌ی درخواست‌ها `reasoning: { effort: "low" }` را از طریق یک `fetch` سفارشی تزریق کن تا پاسخ‌ها پایدار و ارزان بمانند.

## بخش اول: هسته‌ی AI (سنگ‌بنای همه‌ی ایجنت‌ها)

یک ماژول `src/lib/ai.ts` بساز با دو کمکی اصلی:

- **`runAgentText`** — خروجی متنی آزاد (فقط نویسنده از این استفاده می‌کند، چون مصرف‌کننده‌ی خروجی‌اش انسان است نه ایجنت بعدی).
- **`runAgentJSON`** — خروجی ساخت‌یافته: JSON را از پاسخ استخراج کن، با **Zod اعتبارسنجی** کن، و اگر خراب بود **خطا را به مدل برگردان تا یک بار اصلاح کند** (الگوی validate + retry).

> **چرا `generateObject` نه؟** چون بین ده‌ها مدل OpenRouter، پشتیبانی از JSON mode و tool-calling ناسازگار است. استخراج دستی + Zod + retry روی همه‌ی مدل‌ها کار می‌کند و مکانیزمش هم شفاف آموزش داده می‌شود. (در پروژه‌ی واقعی با یک provider مشخص، `generateObject` تمیزتر است.)

تنظیمات مدل:
- مدل پیش‌فرض همه‌ی ایجنت‌ها: `PIPELINE_MODEL` یا `google/gemini-2.5-flash`.
- نویسنده می‌تواند مدل قوی‌تر جدا داشته باشد: `WRITER_MODEL` (مثلاً `anthropic/claude-sonnet-4.5`).

یک فایل `src/lib/company.ts` هم بساز که **زمینه‌ی مشترک همه‌ی ایجنت‌ها** باشد: `COMPANY_PROFILE` (هویت، خدمات چهار رکن، مخاطب هدف بلاگ، هدف تجاری = تبدیل خواننده به «درخواست مشاوره‌ی رایگان») و `BRAND_VOICE` (لحن از برند گاید). این را به system prompt همه تزریق کن؛ وگرنه هر ایجنت تصویر متفاوتی از شرکت می‌سازد.

## بخش دوم: هشت ایجنت (هر کدام یک شغل)

قرارداد خروجی همه را در `src/lib/agents/types.ts` با اسکیماهای Zod تعریف کن. **خروجی هر ایجنت، ورودی ایجنت بعدی است** — این قرارداد همان کاری را می‌کند که interface در کد معمولی.

| # | ایجنت | ورودی | خروجی | دما | نکته‌ی طراحی |
|---|-------|-------|-------|-----|---------------|
| ۱ | **ایده‌یاب** (`idea-scout`) | پروفایل شرکت + عنوان پست‌های قبلی | ≥۳ ایده‌ی امتیازدهی‌شده (عنوان، زاویه، نیت جستجو، امتیاز ۰-۱۰، دلیل) | **۰.۸** | دمای بالا برای تنوع؛ عنوان‌های قبلی را می‌گیرد تا تکرار نکند |
| ۲ | **استراتژیست** (`strategist`) | بهترین ایده | بریف محتوا (مخاطب، کلیدواژه‌ی اصلی/فرعی، outline، تعداد کلمه‌ی هدف، CTA) | ~۰.۷ | جداکردن «تصمیم» از «تولید» |
| ۳ | **پژوهشگر** (`researcher`) | بریف | فکت‌ها، مثال‌ها، سؤالات رایج، یادداشت زاویه | متوسط | الگوی «مدل تصمیم می‌گیرد، کد اجرا می‌کند»: LLM کوئری می‌سازد، `fetch` جستجو می‌کند (**Tavily اختیاری**) |
| ۴ | **نویسنده** (`writer`) | بریف + پژوهش | مقاله‌ی کامل مارک‌داون | ۰.۷ (بازنویسی ۰.۵) | تنها ایجنت با خروجی متن آزاد؛ دو حالت: نگارش اولیه و بازنویسی بر اساس ایرادها |
| ۵ | **ویراستار** (`editor`) | بریف + پیش‌نویس | امتیاز ۰-۱۰۰ + verdict + فهرست ایراد | **۰.۳** | الگوی **Generator/Critic**: نقاد باید از تولیدکننده جدا باشد؛ دمای پایین برای قضاوت پایدار |
| ۶ | **متخصص سئو** (`seo`) | مقاله‌ی نهایی | اسلاگ، متا تایتل/دیسکریپشن، کلیدواژه‌ها، FAQ (برای JSON-LD) | پایین | نیمی LLM (قضاوتی)، نیمی **کد قطعی** (`seo-checks.ts`) |
| ۷ | **ناشر** | همه‌چیز | رکورد پست در store | — | این خودش LLM نیست؛ منطق است (پایین) |
| ۸ | **منتقد** (`critic`) | گزارش کل اجرا + بازخورد انسانی | حداکثر ۳ «درس» خطاب به ایجنت‌های مشخص | ۰.۴ | موتور خودبهبودی |

> **کار قطعی را به LLM نسپار** (`seo-checks.ts`): طول متا دیسکریپشن، تعداد `# H1`، حضور کلیدواژه در عنوان و مقدمه، تعداد کلمات — این‌ها را با **کد معمولی** چک کن، نه با مدل. هرچه قابل‌محاسبه است، به LLM نده.

## بخش سوم: ارکستریتور — مهم‌ترین درس این هفته

فایل `src/lib/agents/orchestrator.ts` جریان کار را اداره می‌کند و **خودش LLM نیست؛ کد قطعی است.**

- «تصمیم‌های خلاقانه» → ایجنت‌ها (LLM)
- «جریان کار» (ترتیب، حلقه‌ی بازبینی، شرط‌ها، ثبت وضعیت) → کد معمولی

> رایج‌ترین اشتباه در ساخت مولتی‌ایجنت این است که orchestration را هم به یک LLM «مدیر» بسپارند. جریان کار باید **قابل پیش‌بینی، قابل دیباگ و قابل تست** باشد — این‌ها ویژگی کد است، نه مدل.

جریان:

```
ایده‌یاب → استراتژیست → پژوهشگر → نویسنده ⇄ ویراستار → سئو → ناشر → منتقد
```

قواعد جریان (با کد، نه مدل):
- **دروازه‌ی کیفیت:** اگر امتیاز ویراستار زیر `APPROVE_THRESHOLD = 75` بود، برگرد به نویسنده برای بازنویسی. حداکثر `MAX_REVISION_ROUNDS = 2` دور. بعد از سقف، تصمیم به انسان واگذار می‌شود.
- **انتشار مشروط (human-in-the-loop):** امتیاز ≥ ۷۵ و verdict = approve → انتشار خودکار (`published`). در غیر این صورت → `draft` تا انسان در استودیو تأیید کند.
- **وضعیت در دیتابیس، نه در حافظه:** هر گام را بلافاصله در store ثبت کن (شروع/پایان/خلاصه/خروجی). استودیو با polling ساده نمای زنده می‌سازد.

## بخش چهارم: مکانیزم خودبهبودی (Self-Improvement)

یک حلقه‌ی بسته‌ی یادگیری **بدون fine-tuning و بدون تغییر کد**:

1. بعد از هر اجرا، **منتقد** کل فرایند را مرور می‌کند (امتیاز ویراستار، تعداد دور بازنویسی، چک‌های سئوی ردشده، خود متن) و حداکثر ۳ **درس** استخراج می‌کند. هر درس خطاب به یک ایجنت مشخص است — مثلاً: *«writer: در مقدمه به‌جای کلی‌گویی، با یک مسئله‌ی ملموس مخاطب شروع کن.»*
2. **بازخورد انسانی** (👍/👎 + توضیح در استودیو) هم از همین مسیر به درس تبدیل می‌شود.
3. درس‌ها در جدول `lessons` ذخیره می‌شوند و `src/lib/agents/lessons.ts` قبل از هر اجرا، درس‌های فعالِ هر ایجنت را به system prompt او **تزریق** می‌کند.
4. **نظارت انسانی روی حافظه:** سقف ۸ درس فعال به‌ازای هر ایجنت (قدیمی‌ها بازنشسته می‌شوند) و امکان حذف دستی درس اشتباه در استودیو — تا خودِ مکانیزم سالم بماند.

## بخش پنجم: مدل داده و لایه‌ی ذخیره‌سازی

**الگوی Adapter:** یک interface `BlogStore` تعریف کن و دو پیاده‌سازی بده:
- `memory.ts` — درون‌حافظه‌ای؛ سیستم **بدون Supabase** کاملاً کار می‌کند (داده با ری‌استارت پاک می‌شود — برای یادگیری کافی است).
- `supabase.ts` — ذخیره‌سازی دائمی. `getStore()` بر اساس وجود متغیرهای محیطی یکی را انتخاب می‌کند.

اسکیمای Supabase (`supabase/schema.sql`) — ۴ جدول:

```
posts          (id, run_id, title, slug unique, excerpt, content_md,
                meta_title, meta_description, keywords jsonb, faq jsonb,
                score, status['draft'|'published'], created_at, published_at)
pipeline_runs  (id, status['running'|'done'|'error'], topic_hint,
                steps jsonb, post_id, error, created_at, finished_at)
lessons        (id, agent, lesson, source['critic'|'human'], active, created_at)
feedback       (id, post_id, rating, comment, created_at)
```

## بخش ششم: رابط کاربری

### ۱. استودیو (`/studio`) — داشبورد کنترل

- دکمه‌ی «شروع 🚀» برای اجرای پایپ‌لاین (با امکان دادن یک topic hint اختیاری).
- **خط تولید زنده:** گام‌به‌گام هر ایجنت با وضعیت و خلاصه‌ی خروجی (polling روی رکورد اجرا).
- تب پست‌ها: لیست همه، تأیید/انتشار پیش‌نویس‌ها (human-in-the-loop).
- تب درس‌ها: مشاهده و حذف حافظه‌ی خودبهبودی.
- قفل اختیاری با `STUDIO_PASSWORD` برای دیپلوی عمومی.

### ۲. بلاگ عمومی (`/blog` و `/blog/[slug]`)

- فهرست مقالات منتشرشده + صفحه‌ی مقاله با رندر مارک‌داون.
- **متادیتای کامل** (title/description از خروجی سئو) و **JSON-LD** (Article + FAQPage از FAQ تولیدشده).
- RTL و برندینگ آرکان.

### ۳. API

```
/api/pipeline/run          ← شروع یک اجرا
/api/pipeline/runs/[id]    ← وضعیت زنده‌ی اجرا (برای polling استودیو)
/api/posts  و  /api/posts/[id]  ← مدیریت انتشار (human-in-the-loop)
/api/feedback              ← بازخورد انسانی → درس
/api/lessons               ← مشاهده/حذف حافظه‌ی خودبهبودی
/api/cron/weekly           ← اجرای خودکار هفتگی (محافظت با CRON_SECRET)
```

## بخش هفتم: استقرار و کرون هفتگی

در `vercel.json` یک کرون تعریف کن (مثلاً دوشنبه‌ها ۶ صبح UTC → `/api/cron/weekly`) تا **هر هفته خودکار یک مقاله‌ی جدید** تولید شود؛ اگر ویراستار تأیید کند مستقیم منتشر می‌شود، وگرنه در استودیو منتظر تأیید انسان می‌ماند. روی Vercel، Supabase اجباری است (حالت حافظه بین اجراهای serverless پایدار نیست).

متغیرهای محیطی:

| متغیر | کارکرد |
|-------|--------|
| `OPENROUTER_API_KEY` | **اجباری** — همان کلید فاز ۲ |
| `PIPELINE_MODEL` | مدل پیش‌فرض ایجنت‌ها (پیش‌فرض `google/gemini-2.5-flash`) |
| `WRITER_MODEL` | مدل قوی‌تر فقط برای نویسنده (اختیاری) |
| `TAVILY_API_KEY` | جستجوی واقعی وب برای پژوهشگر (اختیاری) |
| `NEXT_PUBLIC_SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` | ذخیره‌سازی دائمی |
| `STUDIO_PASSWORD` | قفل استودیو |
| `CRON_SECRET` | محافظت endpoint کرون |

## الزامات کیفی

- کد ماژولار: هسته‌ی AI جدا، هر ایجنت یک فایل، ارکستریتور جدا از ایجنت‌ها، store پشت interface.
- کامنت‌های آموزشی فارسی روی تصمیم‌های کلیدی (چرا Zod؟ چرا ارکستریتور کد است؟ چرا سئو نیمه‌قطعی؟).
- کلیدهای API فقط سمت سرور (`server-only`).
- مدیریت خطا در هر گام؛ خطای یک ایجنت کل اجرا را با پیام روشن متوقف کند و در store ثبت شود.
- RTL و دسترسی‌پذیری کامل در UI طبق برند گاید.

## ترتیب کاری که از تو می‌خواهم

1. دو فایل brief و brand-guide را بخوان؛ خلاصه‌ی درکت را بده.
2. هسته‌ی AI (`ai.ts`) + `company.ts` + قرارداد خروجی (`types.ts`) را بساز.
3. ایجنت‌ها را یکی‌یکی بساز و هر کدام را مستقل تست کن (ایده‌یاب → … → منتقد).
4. ارکستریتور را بساز و حلقه‌ی بازبینی و دروازه‌ی کیفیت را در آن پیاده کن.
5. لایه‌ی store (اول memory، بعد supabase) و مکانیزم درس‌ها را وصل کن.
6. استودیو، بلاگ عمومی، و API routeها.
7. کرون هفتگی و آماده‌سازی استقرار.

در هر مرحله مختصر توضیح بده چه ساختی و بعد ادامه بده. اگر جایی تصمیم مهمی لازم بود، اول از من بپرس.
